# Brainstorming — 아이디어를 설계로

목적: 모호한 아이디어를 대화로 좁혀 **DESIGN.md**(승인된 설계 문서)를 만들고,
/write-plan 절차로 넘긴다. 이 스킬이 활성인 동안 **구현 금지, PLAN.md 생성 금지.**

## 절차

### 1. 컨텍스트 파악 (질문 전에)
프로젝트의 기존 파일·문서·최근 커밋을 먼저 훑는다. 빈 프로젝트면
사용자의 다른 프로젝트 관례(스택, 구조)를 물어볼 준비를 한다.

### 2. 질문으로 좁히기
- **한 번에 질문 하나.** 여러 개를 쏟아내지 않는다.
- **객관식 우선.** 열린 질문보다 "A/B/C 중 어느 쪽?"이 답하기 쉽다.
  열린 질문은 객관식으로 못 만들 때만.
- 커버할 것: **목적**(누가 왜 쓰나) → **핵심 시나리오**(가장 중요한 사용 흐름
  1~3개) → **제약**(스택, 기존 시스템 연동, 예산/기간) → **성공 기준**
  (뭐가 되면 완성인가) → **비범위**(안 만들 것 — 명시적으로).
- 답이 이미 대화/컨텍스트에 있으면 묻지 말고 확인만 한다.

### 3. 접근법 제시
이해가 됐다고 판단되면 **2~3개 접근법을 트레이드오프와 함께** 제시하고,
추천안을 이유와 함께 앞세운다. 사용자가 고르면 진행.

### 4. 설계 섹션별 확인
설계를 한 번에 다 쓰지 말고 섹션 단위로 제시하고 각각 확인받는다:
아키텍처 → 컴포넌트 → 데이터 흐름 → 에러 처리 → 테스트 전략.
단순한 섹션은 몇 문장, 복잡한 섹션은 200~300단어까지. 확인 중 어긋나면
앞 단계로 돌아가는 것을 주저하지 않는다.

### 5. DESIGN.md 작성
승인된 내용을 프로젝트 루트 DESIGN.md로 저장:

```markdown
# DESIGN: [이름]
## 목적 / 사용자
## 범위 / 비범위
## 접근법 (선택 이유 + 기각된 대안)
## 아키텍처 / 컴포넌트
## 데이터 흐름
## 에러 처리
## 테스트 전략
## 미해결 (있다면 — write-plan의 DECISIONS 후보)
```

저장 전 자기 점검: 플레이스홀더·상호 모순·비범위 침범이 없는지 확인하고 고친다.

### 6. 핸드오프
DESIGN.md 승인 후 **/write-plan 절차로 이어간다** (DESIGN.md를 입력으로
PLAN.md 생성). 대형/고위험 설계면 write-plan 전에 grill 스킬로 심문을 제안한다.

## 원칙
- **YAGNI 무자비하게** — 모든 접근법에서 불필요 기능을 걷어낸다.
- 프로젝트가 한 설계로 감당 안 되게 크면 서브프로젝트로 분해를 돕고,
  첫 서브프로젝트부터 이 절차를 다시 돈다.
- 사용자가 "그냥 만들어"라고 재촉해도, 목적·성공 기준·비범위 3개는
  확보한 후에 넘어간다 (최소 안전선).
