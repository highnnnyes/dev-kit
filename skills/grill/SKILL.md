# Grill — 설계/계획 심문

목적: 승인 직전의 설계나 계획을 압박 검증해서, 구현 중에 터질 문제를
구현 전에 드러낸다. **문서가 이미 있어야 작동한다** — 없으면 brainstorming으로.

## 절차

### 1. 대상 파악
DESIGN.md 또는 PLAN.md(지정된 문서)를 읽고, 관련 코드가 있으면 함께 확인한다.
문서의 주장과 코드의 실제가 다른 지점을 먼저 수집한다.

### 2. 심문 — 한 번에 하나씩
아래 각도에서 약점이 큰 순서로 질문한다. **질문마다 추천 답을 함께 제시**해서
세션이 앞으로 굴러가게 한다 (질문만 던지고 막지 않는다):

- **숨은 가정**: "이 설계는 X가 항상 참이라고 가정하는데, Y인 경우는?"
- **의존 사슬**: "3단계가 실패하면 1~2단계 산출물은 어떻게 되나?"
- **경계/에러 경로**: 빈 입력, 동시 접근, 부분 실패, 재시도.
- **규모**: 데이터/사용자가 10배가 되면 어디가 먼저 깨지나?
- **되돌리기**: 이 변경을 롤백해야 하면 가능한가? 마이그레이션은?
- **비범위 침범**: 계획에 설계의 비범위 항목이 슬쩍 들어와 있지 않나?
- **verify 실효성**: 각 태스크의 verify가 실제로 그 목표를 검증하나,
  아니면 "돌아가기만 하면 통과"인가?

### 3. 판정 기준
- 사용자가 답하지 못하는 질문 = 설계 구멍. 그 자리에서 함께 답을 만들거나
  DECISIONS/미해결 항목으로 문서에 추가한다.
- 확신이 낮은 지적으로 시간을 끌지 않는다 — 치명도 높은 것부터,
  세션은 5~10개 질문 내로 끝낸다.

### 4. 산출물
심문이 끝나면 대상 문서를 갱신한다:
- 드러난 가정 → 설계에 명시 또는 DECISIONS로 승격
- 발견된 실패 모드 → 해당 태스크의 verify 강화 또는 새 태스크 추가
- 판정 요약을 문서 하단에 기록:
  `## Grilled: [날짜] — 수정 N건, DECISIONS 추가 M건`

### 5. 핸드오프
- 대상이 DESIGN.md였으면 → /write-plan 절차로.
- 대상이 PLAN.md였으면 → 갱신된 계획으로 /execute-plan 절차로.

## 언제 쓰나 / 안 쓰나
- **쓴다**: 여러 시스템에 걸친 설계, 데이터 마이그레이션, 결제·인증 등
  고위험 도메인, 되돌리기 어려운 결정이 포함된 계획.
- **안 쓴다**: 소형 프로젝트, 단일 파일 기능, 이미 여러 번 해본 패턴 —
  이런 데 심문은 의식(儀式)일 뿐이다.
