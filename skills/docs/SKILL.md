# docs — 개발자용 기술 문서 스킬

이 워크플로우에서 기술 문서의 1차 독자는 **에이전트**, 2차 독자가 사람이다.
builder는 태스크마다 신선한 컨텍스트로 뜬다 — 시스템 구조를 매번 코드
탐색으로 재구성하면 느리고 비싸고 틀린다. 좋은 기술 문서는 그 탐색을
대체하는 **압축된 컨텍스트**다. 사람에게 좋은 문서와 에이전트에게 좋은
문서는 같은 문서다: 짧고, 정확하고, 현재 상태와 일치하는 것.

## 문서 4종과 각자의 질문

| 문서 | 답하는 질문 | 갱신 시점 |
|---|---|---|
| `ARCHITECTURE.md` | 시스템이 어떻게 생겼나 — 모듈 맵, 데이터 흐름, 경계 | 구조 변경 stage 완료 시 |
| `docs/api.md` (또는 스키마 파일) | 모듈/서비스 간 계약이 정확히 뭔가 | 인터페이스 변경 시 |
| `docs/data-model.md` | 데이터가 어떤 모양이고 왜 그런가 (ERD, 스키마, 제약) | 스키마 변경 시 |
| `docs/decisions/NNN-제목.md` (ADR) | 왜 이렇게 결정했나 — 대안과 트레이드오프 | 결정 시점에 1회, 이후 불변 |

**기존 산출물과의 경계** — 겹치면 안 된다:
- `DESIGN.md` = 기능 하나의 설계 스냅샷 (brainstorming 산출물, 시간이 지나면 낡아도 됨)
- `PLAN.md` / `PROGRESS.md` = 무엇을 할지 / 했는지 (실행 기록)
- **이 스킬의 문서 = 시스템의 현재 상태** (살아있는 문서, 낡으면 즉시 갱신 대상)
- ADR만 예외: 결정의 역사라서 불변. 뒤집을 땐 새 ADR로 이전 것을 supersede.

## 철칙 — 문서 slop 금지

1. **코드가 답할 수 있으면 쓰지 않는다.** 함수 시그니처 나열, 파일 트리
   복사, "이 클래스는 X를 담당한다" 수준의 나열 금지. 문서의 존재 이유는
   코드에서 안 보이는 것: 모듈 **사이**의 흐름, 경계의 **이유**, 결정의 **맥락**.
2. **에이전트 독자 테스트.** 작성 후 자문: "신선한 builder가 이것만 읽고
   태스크에 착수할 수 있나?" 못 하면 부족한 거고, 코드 읽는 게 더 빠르면
   과잉인 거다.
3. **살아있는 문서는 짧게 유지한다.** ARCHITECTURE.md는 200줄 이하.
   길어지면 갱신이 밀리고, 갱신 밀린 문서는 에이전트를 오도한다 —
   틀린 컨텍스트 주입은 컨텍스트 없음보다 나쁘다.
4. **다이어그램은 텍스트 기반만** (mermaid, ASCII). 이미지는 에이전트가
   못 고치고, 못 고치는 다이어그램은 반드시 낡는다.

## 절차

### 1. 현황 파악
- 문서 4종의 유무와 마지막 갱신 시점 확인 (git log)
- **drift 스캔**: 문서의 주장 vs 코드의 실제
  - ARCHITECTURE.md의 모듈 목록·의존 방향 vs 실제 import 그래프
  - api.md의 엔드포인트/시그니처 vs 실제 라우트·타입 정의
  - data-model.md의 스키마 vs 실제 마이그레이션/모델 파일
- drift 발견 → 신규 작성보다 갱신이 항상 우선

### 2. 프로젝트 크기별 스코프
- **소형 (파일 <20개)**: ARCHITECTURE.md 하나에 4종 압축 (구조 + 주요
  인터페이스 + 데이터 모양 + 결정 몇 줄). 파일 쪼개기는 과잉.
- **중형 (esi-flow류)**: ARCHITECTURE.md + ADR. api/data-model은
  해당 복잡도가 생길 때.
- **대형/다인원**: 4종 전부 + 개발 환경 셋업(`docs/setup.md`).

### 3. 작성 기준
- **ARCHITECTURE.md 구조**: 한 문단 개요 → 모듈 맵(mermaid) →
  핵심 데이터 흐름 1~3개(요청이 들어와서 나가기까지) → 경계 규칙
  ("A는 B를 직접 부르지 않는다, 이유: ...") → 확장 포인트
- **ADR 형식**: 상태(제안/채택/폐기) / 맥락 / 결정 / 근거 / 기각한 대안 /
  결과. 10줄이면 충분 — 논문 금지.
- **api.md**: 실제 코드에서 추출 가능하면 추출을 우선 (OpenAPI, 타입
  정의 링크). 손으로 옮겨 적는 명세는 drift 1순위다.

### 4. 검증 (verify)
- drift 스캔 재실행: 작성/갱신한 문서가 현재 코드와 일치
- mermaid 문법 유효성
- 에이전트 독자 테스트를 reviewer가 수행: "이 문서로 다음 stage 첫
  태스크에 착수 가능한가"를 PASS 조건에 포함

## 루프 통합

1. **write-plan**: 계획에 구조 변경(새 모듈, 인터페이스 변경, 스키마
   변경)이 포함되면 해당 stage 마지막에 문서 태스크 자동 추가:
   ```markdown
   - [ ] N.x ARCHITECTURE.md 갱신 · role: docs · verify: drift 스캔 통과
   ```
   구조 변경 없는 stage에는 넣지 않는다 — 매 stage 문서 갱신은 과잉.
2. **DECISIONS → ADR 파이프**: write-plan의 DECISIONS에서 하이가
   결정을 내리면, 그 결정+근거를 `docs/decisions/`에 ADR로 자동 기록.
   결정의 맥락이 채팅에서 증발하는 걸 막는다.
3. **builder 브리핑**: 오케스트레이터는 builder 투입 시 ARCHITECTURE.md
   존재를 브리핑에 명시 — "구조 파악은 이 문서 먼저, 코드 탐색은 그다음".

## 트리거 정리
- "개발문서/아키텍처 문서/API 명세/ERD 만들어줘" → 즉시 이 스킬
- write-plan에 구조 변경 stage 존재 → 문서 태스크 자동 포함
- DECISIONS 결정 확정 → ADR 자동 기록
- "문서 검사해줘" → drift 스캔만, 수정은 보고 후
- 새 프로젝트 첫 구조 확정 (brainstorming의 DESIGN.md 승인 직후) →
  ARCHITECTURE.md 초판 생성
